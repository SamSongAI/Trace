# FlashNote macOS MVP Design (2026-02-27)

## 1. Architecture Overview
FlashNote MVP is implemented as a native macOS SwiftUI app with an AppKit shell for system-level capabilities. The app runs as a menu-bar utility (`NSStatusItem`) and keeps no main document window. Core responsibilities are split into four layers. `AppDelegate` owns lifecycle, menu-bar actions, and global hotkey registration. `CapturePanelController` owns the floating capture panel and interaction orchestration (show/hide, keyboard handling, pin behavior, focus). `DailyNoteWriter` is the storage engine for deterministic writes into Obsidian Daily Notes. `AppSettings` centralizes persisted user configuration in `UserDefaults`, including vault path, file-name format, hotkey mapping, and panel frame persistence. This separation isolates the file write algorithm from UI concerns so behavior can be tested or replaced without refactoring panel code. The design prioritizes low-latency capture over feature breadth: no cloud, no sync, no plugin dependency, no background server. The app remains local-first and file-system-native. The architecture also intentionally keeps all external dependencies at zero for MVP stability and distribution simplicity.

## 2. Component & Data Flow
Interaction flow starts from the globally registered hotkey (`GlobalHotKeyManager`). When pressed, the app toggles the floating panel. Panel content is rendered by `CaptureView` with `CaptureViewModel`, where the user enters text, changes section (Note/Clip/Link/Task/Project), and toggles Pin mode. `Cmd+Enter` triggers save. Save requests pass to `DailyNoteWriter` with payload `{text, section, now}`. Writer resolves target path using `vaultPath + dailyFolder + formattedDateFileName`, creates the daily folder if needed, and loads existing file (or generates initial template with all section headers). It ensures all required section headers exist, then inserts the new code-block entry immediately under the selected header with timestamp (`yyyy-MM-dd HH:mm`) and writes file back in UTF-8 atomically. On success: if pinned, clear input and keep panel visible; otherwise close panel and return user to current context. On failure (missing vault path, write issue), panel shows a modal warning sheet and preserves user text.

## 3. Error Handling, Testing, and Rollout
Error handling is explicit at service boundaries. `DailyNoteWriter` throws typed errors (for example invalid vault path), and UI controllers convert those errors into user-facing alerts rather than silent failures. Hotkey registration failure is logged and recoverable through settings changes. For testing, MVP validation focuses on deterministic file mutations and critical UX paths: (1) first-run file creation with all headers, (2) insertion under existing header, (3) insertion when header is missing, (4) pin-mode continuous capture behavior, and (5) panel toggling via global shortcut. Given current machine constraints, build verification in this workspace is blocked by local Command Line Tools/SwiftPM mismatch (`PackageDescription` link failure and SDK/compiler mismatch), so code-level validation currently relies on static review and modular boundaries. Rollout plan: first fix toolchain alignment (full Xcode installation and active developer dir switch), then run compile/lint pass, then implement test target for `DailyNoteWriter`, then package app for DMG distribution. This keeps product momentum while isolating environment risk from product logic.
